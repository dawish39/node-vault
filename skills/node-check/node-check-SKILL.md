# node-check — 節點檢核 Prompt 生成器

在 session-wrap-ex 產出知識節點後，從當前對話 context 提取資訊，生成一份結構化檢核 prompt。使用者將這份 prompt 帶到新對話執行獨立檢核，實現生成端與驗證端的 context 隔離。

這樣做的原因：生成節點的對話 context 裡帶有討論過程的 framing，驗證者如果繼承同一個 context，容易受到生成階段的偏差影響。拆到新對話讓驗證者從零開始，只基於節點文字本身判斷。

---

## 防呆檢查（觸發後第一步）

掃描當前 context，確認存在 session-wrap-ex 產出的節點內容（frontmatter + 正文）。

- **找到** → 繼續執行
- **找不到** → 停止，輸出：「這個 skill 需要在 session-wrap-ex 產出節點的同一對話中使用。請確認節點已產出，或提供節點內容。」

---

## 執行流程

通過防呆後，依序完成以下四個提取步驟，然後組裝成一份完整 prompt 輸出。不需要讀取任何檔案，所有資訊都從 context 取。

### Step 1：提取背景資訊

從 context 中提取：

- **節點檔名**：session-wrap-ex Step 3 checklist 裡的檔名
- **節點來源**：frontmatter 的 Source 欄位，或從對話脈絡摘要
- **分析處理方式**：節點經過什麼分析流程（例如 Steelman 檢核、理論對接、事實重建等）
- **核心主張**：節點的 Core insight 或主要結論是什麼

### Step 2：提取外部引用清單

掃描 context 中的節點正文，識別所有外部可驗證引用。

**納入的引用類型**：學術文獻、業界數據、操作手冊、文章摘要、具名人物的引述、具體統計數字與其來源。

**排除的來源**：使用者本人的觀察與經歷、截圖內容、聊天紀錄、對話中的第一手描述。這些是第一手來源，web search 查不到也不該查。

對每條引用，記錄：
- 來源名稱（作者、年份、出處）
- 節點中如何使用這條引用（引用了什麼概念、數據、原話）
- 是原始理論的直接引用，還是受啟發的操作化延伸（如果 context 中能判斷的話）

如果整個節點沒有外部可驗證引用，記錄「無外部引用」。

### Step 3：提取 see_also 清單

從 context 中的 frontmatter 提取 `see_also` 欄位裡的節點清單。

- `see_also` 裡的節點 → 納入 Phase 3 檢核範圍
- `see_also_pending` 裡的節點 → 不納入（指向尚未建立的節點）

如果沒有 `see_also`，記錄「無跨節點引用」。

### Step 4：判斷節點特徵

快速判斷以下兩個特徵（影響 Phase 2 的適用項目）：

- **有沒有 Core insight 區塊**？（影響第 8 條檢查項）
- **Known boundaries 段落存不存在**？（影響第 9 條檢查項）

---

## 路徑解析規則

Prompt 模板中的 `{node_vault_path}` 必須在組裝時替換為使用者當前的 Node_Vault 實際路徑。解析優先順序：

1. 當前對話已掛載的 workspace 資料夾路徑（如果掛載的就是 Node_Vault 或其父資料夾）
2. 從 context 中尋找使用者提過的 Node_Vault 路徑
3. 以上都找不到 → 在 prompt 中保留 `{node_vault_path}` 佔位符，並在 prompt 開頭加一行提示：「請先確認你的 Node_Vault 路徑，將 {node_vault_path} 替換後再執行。」

## Prompt 組裝

將以上提取結果組裝為一份完整的檢核 prompt。下方是模板，`{花括號}` 內為從 context 填入的變數。產出時所有花括號變數必須替換為實際內容，不得殘留佔位符。

產出的 prompt 用一個 markdown code block（三個反引號）包住整段輸出，這樣使用者在聊天介面會看到右上角的複製鍵，一鍵複製整份 prompt。不要在 code block 外面加額外說明或前言，直接輸出。

---

### Prompt 模板

```
我需要你檢核一個知識節點的邏輯完整性與外部引用正確性。

## 節點資訊

**檔案位置**：@{node_vault_path}\Node\{節點檔名}

**背景**：{一到兩句話描述節點來源與經過什麼分析處理}。核心主張：{一句話描述}。

**你的角色**：獨立審查者。不預設節點結論正確，從頭檢核。

**重要邊界**：你看不到這個節點的生成過程（對話脈絡、被放棄的選項、中間討論），只能基於節點文字本身判斷。如果你發現節點裡有明顯缺少脈絡的跳躍（某個結論的前提在節點裡找不到），請標注出來——這可能是生成過程中討論過但沒寫進節點的內容。

**操作指令**：先完整讀完節點，再開始逐項查核。不要邊讀邊查。

---

## Phase 1：外部引用查核

用 Web Search 查證以下每一條引用，每條檢查三個層次：

### 引用 1：{來源名稱}
節點中的使用方式：{具體描述節點怎麼用這條引用——引用了什麼概念、數據、原話，不能只寫來源名稱}
請查核：
1. **存在性**：來源是否真實存在？引用的數據或原話是否正確？
2. **詮釋性**：節點對這個來源的使用是否忠於原意？有無過度延伸或斷章取義？
3. **歸屬誠實度**：這是原始理論的直接引用，還是受啟發的操作化延伸？節點的標記是否誠實反映這個區分？

### 引用 2：{來源名稱}
...（重複至所有引用列完）

每條查核結果標明：✅ 正確（附一句簡短理由）/ ⚠️ 部分偏差（說明哪裡偏、正確內容是什麼）/ ❌ 錯誤（說明正確內容）。三種標記都必須附判定依據。

---

## Phase 2：內部邏輯檢核

逐一檢查以下項目：

1. **邏輯跳躍 / 隱含前提**：A→B 的推論中間是否有未說明的前提？
2. **對立證據遺漏**：有沒有明顯的反例或反論沒被處理？
3. **assumption 標記完整性**：有沒有推測性判斷漏標 `[assumption — unconfirmed]`？已標記的有沒有附推測依據？
4. **宣稱強度 vs 證據匹配**：結論的措辭力度跟支撐它的證據數量與品質匹配嗎？
5. **推論方向性**：是從事實推出結論，還是從結論回頭挑選事實？
6. **事實層 vs 分析層混淆**：事實段落裡有沒有偷渡推論？分析段落的推論能不能追溯到事實段落？
7. **使用者本人作為推論證據**：有沒有用 Dawish 的個人經歷或職涯當作因果推論的證據？如果有，是標記為描述性觀察（可以），還是被當成因果論證（需外部文獻支撐或降級措辭）？（檢查理由：AI 協作產出的節點有一個特有失敗模式——AI 容易用使用者個人經歷當因果證據，使用者因自我肯定效應不容易察覺。）
{如果有 Core insight 區塊，加入：}
8. **Core insight 可複用性 vs 證據基礎**：Core insight 是節點中宣稱最強的一段——它主張一個可跨情境複用的框架。這個可複用性宣稱跟節點提供的證據基礎匹配嗎？泛化風險有沒有在 Known boundaries 裡處理？
{如果有 Known boundaries，加入：}
9. **Known boundaries 完整性**：節點正文中浮現的限制（包含標了 assumption 的判斷、樣本量不足、場景侷限性等），有沒有被 Known boundaries 段落漏掉？

---

## Phase 3：跨節點一致性

讀取以下既有節點，逐一確認：

{列出 see_also 節點清單，每個帶完整路徑：}
- @{node_vault_path}\Node\{see_also 節點 1 檔名}
- @{node_vault_path}\Node\{see_also 節點 2 檔名}
...

檢查：
1. 本節點引用這些既有節點的內容時，有沒有扭曲原意？
2. 本節點的判斷有沒有跟這些既有節點矛盾的地方？
3. 本節點宣稱填補的缺口，在這些既有節點中是否真的不存在（還是已被部分處理但本節點沒注意到）？

---

## 輸出格式

直接在對話中輸出報告，不要建立檔案或產出 MD 文件。

請依以下格式：

### 必修（需修正的錯誤）
影響節點正確性的問題，必須處理。

### 選修（建議強化的弱點）
不影響正確性但影響嚴謹度的問題，可選擇處理。

### 確認無誤
通過檢核的部分，清單列出即可，不用展開。

---

## 執行效率指引

**Phase 1 搜尋策略**：每次搜尋用 3-5 個關鍵字，寧可拆成多次短搜尋也不要一次塞 8-10 個詞的精確搜尋。關鍵字太多會找不到東西。

**Phase 3 讀取範圍**：只讀取 see_also 節點的 frontmatter、Core insight 區塊、Known boundaries 區塊，不需要讀完整份節點。跨節點一致性檢查只需要對比核心主張和概念定義，不需要逐段比對。
```

### Prompt 模板使用規則

- Phase 1 的引用清單要具體到「節點說了什麼」。不能只寫「請查核 Meyer & Rowan」——要寫「節點引用了 Meyer & Rowan (1977) 的 decoupling 概念，宣稱資安部門滿足 decoupling 的觸發條件因為成功不可觀測，請查核這是否為 Meyer & Rowan 原文主張」。
- Phase 1 如果無外部引用，該段替換為：「本節點無外部可驗證引用，Phase 1 不適用，請直接進入 Phase 2。」
- Phase 2 的條目數量根據節點特徵調整（7-9 條）。沒有 Core insight 就不列第 8 條，沒有 Known boundaries 就不列第 9 條。不寫「不適用」，直接不列入。
- Phase 3 如果無 see_also，該段替換為：「本節點無 see_also 引用，Phase 3 不適用。」
- Phase 3 的節點路徑使用完整路徑格式，確保新對話的 Claude 能直接讀取。
